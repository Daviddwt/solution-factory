Codex Plugin · Proposal Workflow

把方案 PPT 的前半段，做成一条可复用生产线。

Solution Factory 把客户资料、公司能力、需求拆解和逐页 PPT 脚本放进同一条流程里。它不承诺“一键生成最终 PPT”，只把最容易乱的前置工作变成可审阅、可追溯、可交接的标准产物。

Installation Strategy

推广时不要教人敲命令，只给 Codex 一句话。

这才是面向同事和外部用户的低摩擦入口：让 Codex 自己读 README、找 Release、下载安装、启动网页工作台并验证。

请给我的 Codex 安装以下地址的插件：https://github.com/Daviddwt/solution-factory
请自行读取注意事项、下载最新分享包、安装插件、打开网页工作台并完成验证。
把复杂安装藏到 Codex 后面

对外只传播结果和入口，不传播终端细节。失败时也让 Codex 检查安装、服务、端口和网页。

Why It Matters

售前方案不是缺 PPT 工具，而是缺一套可靠的前置生产秩序。

适合在 X 上讲的不是“我做了个工具”，而是“我把一个混乱工作流拆成了可复用方法”。

资料很多，口径很散

客户材料、产品手册、案例、会议纪要分散在不同地方，直接做 PPT 很容易遗漏和乱承诺。

需求和能力边界混在一起

哪些是现有产品、哪些要集成、哪些要定制，必须先拆清楚，否则越到后面返工越大。

图片 PPT 不是源头

真正的源头应该是需求包、能力匹配表和逐页脚本。图片生成只负责表达，不负责替你判断事实。

Production Line

从“资料堆”到“可审脚本”，中间必须有 6 个标准环节。

这套流程适合推广，因为它让同事知道：工具不是替你拍脑袋，而是把每一步的输入、输出和审核点显性化。

01材料归集

客户资料、公司知识库、产品 FRS、参考 PPT 风格。

02需求拆解

场景、角色、痛点、目标、流程、数据、接口。

03能力匹配

标注 existing、configurable、integration、custom_dev、unclear。

04实现路径

产品模块、开发项、流程设计、数据流和阶段计划。

05逐页脚本

标题、核心观点、讲稿、图示类型、证据引用。

06图片提示词

交给 NotebookLM 或图片 PPT 智能体，作为渲染层执行。

Product Proof

它有网页工作台，不只是一个提示词文档。

推广页必须让人看到真实界面。截图负责建立可信度，文案负责降低上手门槛。

对同事怎么讲

不要解释依赖、脚本和端口。只说“把 GitHub 地址发给 Codex，它会安装并打开网页”。

对外部用户怎么讲

这是一套方案生产方法，不是万能 PPT 生成器。用户需要提供自己的材料和知识库。

失败时怎么讲

网页打不开时，把错误交给 Codex 排查安装、服务进程、端口占用和日志，不让新用户自己猜。

Truthful Boundary

推广可以有锋芒，但边界必须说实话。

这一段要放在页面中部，而不是藏到 FAQ。可信的 AI 工具推广，先讲清楚什么能做，什么不能说成已经接入。

已经验证

需求梳理、能力匹配、逐页 PPT 脚本、图片生产提示词、网页工作台和 Release 分享包。

后续流程

图片 PPT 生成和图片 PPT 转可编辑 PPTX 是下游流程，不能把脚本生产台包装成最终交付器。

敏感资料

公开 GitHub 不放合同正文、报价明细、个人信息、客户项目路径和公司内网地址。

X Writing Strategy

推广文章不要写成说明书，要写成一条问题驱动的 thread。

先讲痛点，再讲方法，再给 GitHub 地址。少说“强大”，多说“我把哪个工作流标准化了”。

1

Hook
开头只讲一个强问题：做方案 PPT 最难的不是排版，而是需求、能力和证据没拆清楚。

2

Method
把生产线画出来：资料归集 -> 需求拆解 -> 能力匹配 -> 逐页脚本 -> 图片提示词。

3

Proof
放网页工作台截图和 GitHub Release 链接，证明它不是空想流程。

4

Boundary
主动说明不是最终 PPT 生成器，只生产脚本和提示词。这个反而增加可信度。

Post 1

做方案 PPT 最难的部分，往往不是排版，而是前面那一大团混乱资料：客户需求、产品能力、案例、接口、待确认事项，全都混在一起。

Post 2

我把这部分做成了一个 Codex 插件：Solution Factory。它不是“一键生成最终 PPT”，而是把 PPT 前置生产线标准化。

Post 3

流程是：客户资料 + 公司知识库 + 风格参考 -> 需求拆解 -> 能力匹配 -> 实现路径 -> 逐页 PPT 脚本 -> 图片生产提示词。

Post 4

最重要的是能力边界：现有产品、可配置、需集成、定制开发、不明确，都要显式标注，不能把没有证据的能力写成默认现有能力。

Post 5

安装也尽量简单：把 GitHub 地址给 Codex，让 Codex 自己读 README、下载 Release、安装插件、启动网页工作台并验证。

Post 6
